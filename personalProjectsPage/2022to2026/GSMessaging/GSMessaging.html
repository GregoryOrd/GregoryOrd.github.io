GSMessaging - Messaging App Exposed Outside the Home Network (May 2025)

Project Description

The github repo for this project is here: https://github.com/GregoryOrd/GregMessaging

This project was intended to be a lightweight messaging app for messages between myself and my partner. She had recently began a new job writing C++ firmware for a company that had many good hardware engineers, but not many software/firmware people. Sometimes when she got stuck, she would send me a text message to see if I could help.

The purpose of this project, was to create a web based messaging app so that we could communicate via the browser on our computers, rather than picking up a cell phone to send a text message. Sending messages from the device I was already using, and being able to use a mouse and keyboard, was less disruptive to my workflow.

This project was built relatively quickly over the course a few weekends. Starting points for both the server and index.html files were generated by an LLM.

The backend was a C++ webserver, built using my GregBuild build system (another personal project from 2021). The webserver ran on a Raspberry Pi Zero behind an Nginx reverse proxy, which was used for TLS termination. The front end was an HTML page using VueJs. All messages were stored in memory, so nothing persisted very long. The Raspberry Pi was plugged/unplugged once a day to restart it and prevent the message list from growing too large. Since I work from home and the Raspberry Pi was nearby I could also power cycle it during the day if anything was acting strangely.

To make the server accessible from outside the home network, I setup portforwarding from our home network access point to the raspberry pi on a static IP. I also got a DNS name setup from a free provider, and setup TLS certificates with Let's Encrypt.

After a few months of this project running, the amount we used it slowly decreased. Eventually the DNS provider I was using shutdown, and I wanted to use the raspberry pi are some other things so the project ended.


Project Risks

This project used a subdomain, from a free DNS provider. That is, the domain name for the site was something like gsmessaging.example.com. This was nice because it was free, but since I didn't have control of example.com, it meant that the owner of example.com could take down might site at anytime without warning. After a few months of the project running, we left for a vacation to Europe for 4 weeks, and when we got back the our domain had been taken down. For a more important project, having more control over the deployment of the system would be necessary.

Lessons Learned

To me, the interesting part of this project was exposing a device from the home network to the internet. While I don't have any concrete lessons learned from this project, it did get me thinking about security concerns that might come along with exposing a device to the internet from a home network. Since the project's lifespan was so short, I didn't try any other setups, but I did spent some time thinking about setups that might improve the security positioning of the system. One idea was to parition the home network somehow to isolate the exposed device. That way, if the exposed device was compromised it wouldn't be able to reach any other devices on the home network. Perhaps this home network paritioning could have been done with some router software configuration, or perhaps adding another switch or router to the home network. This is something I have also considered since, in projects like the Pi Hole Access Point project. Having a more built up home network setup might be something I consider in the future if I decide implement more projects like this.
