Housing Market Monitor - Web Scrapper Monitoring Properties for Sale (Dec 2025 to Jan 2025)

Project Description

The github repo for this project is here: https://github.com/GregoryOrd/HosuingMarketMonitor

This project is a python web scraper that monitors property listings with various search criteria. The search criteria are encoded as query parameters into the URL, and the property website returns an HTML page including javascript scripts that can be loaded to get JSON objects containing the data for all listing found with the given criteria. The property listing website also uses pagination so the code for this project sends HTTP requests until all pages are consumed and parses the JSON objects from each page to write the details of the listings to a database.

Once the listings are in the database, a python webserver (using the fastapi library) server HTML to the user who can access the UI from a browser to see tables with the listings. The user can specify search criteria to filter the listings in the database and can leave notes on each listing or mark the listings as liked, disliked, or neutral.

In additon to the web user interface, an email notification interface was also created. After each run of the web scrapper an email is sent to email account specified in a configuration file to alert them of any new additions, deletions, or modifications to the listings for the given area (town/city). If multiple areas are being monitored, each email account will receive one email per area.

Future Work

At the time of writing, work has stopped on this project but there are still some interesting things that could be done. The intention for this program was to be run repeately to monitor the listings over time. If statistics were calculated and the project was run on a regular interval for enough time, some interesting trends might appear. Some example statstics to be tracked might be: number of listings, average time on market for current listings, average price for certain categories of properties, number of listings in each category of property, percentage of listings in each category, etc.

Project Risks

If the project is to continue for the long term, using webscrapping for the data input does create a situation where the data input process can break at any time. If the maintainers of the website change how they are presenting the data, they can do so without consulting or informing you and your system can break without warning. This can potentially require a significant amount of rework to handle their changes. After a few months away from this project, when I tried to get it up and running again to complete this write up an unannounced interface change that broke my application is exactly what I found.

In many ways, this is similar to the risk situation to the GSMessaging project. In both of these projects, the main risk is that our ability to deploy/run the code properly is dependent on other third parties that we are not in contact with and may change the support the provide to use at any moment without consultation or warning to us.

Lessons Learned

  • Many modern web applications use NextJS
  • NextJS pages often include JS files with src attributes like "src="/_next/static"
  • Web Scrapping Interfaces can be brittle
  • Not having enough control of the entire system can lead to project risk
