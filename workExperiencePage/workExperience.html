The Gateways Team

The team I was on at Calian was a small team called the "Gateways" team. We had various projects, but they were all related to the operation of Satellite Ground Gateways. In particular, our gateways handled aeronautical data and the main purpose of our systems was to connect the ground network to the satellite network. ACARS (Aircraft Communications Addressing and Reporting System) and ADS-B (Automatic Dependent Surveillance - Broadcast) data are some example of the types of data that went through these gateways. Having redundant systems was a common theme for these projects and we had servers at multiple sites around the world.

Features I Lead

While much of the work I did for Calian was with the help of team members, these were some of the tasks where I took the lead:

Adding Support for Using a non-default Java Tuststore and Keystore for Outgoing TLS Interfaces.

For one project, I was tasked with adding support for using a non-default truststore and keystore for outgoing TLS interfaces. These interfaces included connecting to an LDAP server and various database servers. The client libraries we were using already had support for TLS, and the code change to use a non-default truststore and keystore was quite trivial. Where this became interesting was in testing. To test, I needed to configure all of the test databases to require TLS connections, and I needed to generate and configure TLS certificates. There was nothing revolutionary in this task, but it was a good opportunity to learn a little bit about TLS certificates and how java handles TLS.

Designed and built a DNS log reader program

This program read from the log file of the named daemon, parsed out the relevant DNS log data, and wrote it to our postgres database. The postgres table was then displayed in a GUI using code that Calian had already developed. For this program I used the logrotate tool and install pre/post rotate scripts to detect rotations in the named.log file. The program was written in python and used linux signal handlers to receive signals of the logrotatation events from the pre/post rotate scripts. The state of the DNS log reader was also maintained in sqlite DB files so that the program could behave properly on start and restart and could handle down time without any loss up until a reasonable duration of time. To control execution of the program we setup a service file and ran it with systemd.

Implemented a feature to read syslog messages from a FortiOs log to track VPN sessions statistics

Wrote a syslog parser for both the RFC 5424 syslog standard and the BSD syslog standard. Then implemented the logic to track FortiOS VPN tunnel sessions and calculate statistics. Some of the statistics included the current number of active VPN tunnels, the number of successful/failed VPN tunnel setups in a time window, and the average setup time of the successfully setup VPN tunnels.

Setup a Grafana dashboard configuration to show the above VPN statistics as well as other important statistics

For this I wrote a python script to automate the setup and update of the Grafana dashboards and datasources. I also developed a couple of Grafana Datasource plugins to display the data in the ways we needed.

Wrote a C program to redirect UDP traffic from a raw socket to a unix socket

The purpose of this was that we were analyzing traffic from a client's system like a packet sniffer. We needed to know the true source and destination IP addresses, and not inject the IP address of our machine as the destination or the IP address of our client's network monitoring machine as the source. So instead of receiving the data over a regular socket, we read it from a raw socket and then redirected it to our Java application code using a unix socket. The C program that we wrote to do this had an accompanying systemd service file so that it would be turned off in a test environment.

Worked on an operating system upgrade from one version of RHEL to a newer version

This involved some of the following tasks:

  • Using a base image virtual machine to download all of the .rpm files needed by the server from the package manager.
  • For packages that couldn't be downloaded through the package manager, download the source code and build it ourselves. Then used Red Hat's rpmbuild tools to build a .rpm file ourselves.
  • One of these packages that needed to be built from source was python2. We decided to stick with python2 because we didn't want to update some legacy C code that was using the Python2 C Api to embed a python interpreter into a simulator. This was signficant because the new version of RHEL came with python3 and we didn't want to distrupt any operating system services that needed python3 when we installed python2. So the python2 rpm package that we built needed to behave nicely installed side-by-side with python3.
  • Turn the directory of rpm packages into an offline yum repository.
  • Create two SELinux policy modules to allow some of our systemd services to run on the new operating system. For the sake of confifientiality, I won't say what the SELinux policy modules did, but this part of the OS upgrade configuaration did give me the chance to play around with some of the selinux tools and create two basic SELinux policy modules.
  • Make some code changes to our application code to be compatible with updated API in .so libraries and .jar files. For example, upgraded versions of OpenSSL.
  • Update configuration for changed linux services. For example, changing the service used for syncing with the time server changed when upgrading to a new version of REHL. The firewall service that was used also changed so those configurations also needed to be updated.
  • Create a custom .iso file that has all of our packages, SELinux modules, and configurations updates baked into it. To do this we mounted the file system from a base .iso file, then made a copy and overlaid our changes on top of it. To customize the install we used Red Hat kickstart files. Calian had some tools setup to build this .iso file, so I didn't have to setup the entire process, but some of the scripts needed to be updated along side our updated offline rpm package repo, configuration files, and kickstart files.

Support Tasks

In addition to the above tasks, my role at Calian sometimes involved support tasks and bug fixes for issues that came up at the various production sites. These tasks typically involved reading through log data, analyzing messaging sequences, and trying to debug the systems from a distance. In some instances we even created tools to capture data for debugging at a later date. One example was creating a python program that used inotify to monitor a directory that was receiving files from an SFTP transfer and stream a copy of the files into a tarball in another directory before the server's application code could process and delete them. This program allowed us to capture the files and analyze them as part of the debugging process for the particular issue that was happening in production.

Calgary Counselling Centre Group Testing App (May to August 2019) (In Office)

Project Description

The goal of this project was to build an application to allow clients in group counselling programs to complete assessment questionnaires each week throughout a 10-14 week program duration. The application was web-based and was presented to clients via tablets connected to the Calgary Counselling Centre's internal network. This project was completed by myself and another summer student using C# and the .NET framework.

The Calgary Counselling Centre runs 12 different group programs for clients with different needs. Some of the programs are for adults, while others are intended for children. Each group has a "Pre-Package" which is completed at the first session, a "Post-Package" which was completed at the final session, and a standard package that is completed at every session in between. Each package consists of a variable number of individual questionnaires. The application needed to be able to get the client's log-in information, determine which questionnaire package the client was to complete that day, then serve that package to the client. The client's questionnaire answer data was then stored in a database before being pulled out and processed by a software used by counsellors and the centre's Research department. In total, there were about 40 individual questionnaires arranged into approximately 30 packages.


Project Add-ons

In addition to the main project requirements, my partner and I had and implemented some ideas to make the project more complete. The first was an editing interface that allowed users to modify the contents of programs, packages, sections and questions. This functioned through a web API. The second add-on was an automated emailing system. This system detected common database configuration errors and sent an appropriate email to the Centre's IT Coordinator in response. The system was also capable of sending summary reports of important questions, but this feature was turned off. Finally, simple interface was built with another web API to allow retrieve client entered question data. The interface allowed the user to specify filters such as date, program type, package type, client, etc. On the client-side, the retrieved data was formatted into an excel spreadsheet that could be downloaded and inputted into the software used by the Research Department.

I also learned a lot about teamwork in software development. I learned that when designing a system it is important to design for the future as well as for the present moment...and I also learned a little bit about the .NET framework, C#, SQL, databases, HTML, javascript and CSS.
