While working on one legacy project, I wanted to setup a better IDE. Our typical development environment for most projects was to run a virtual machine with the same version of RHEL as the production servers.

This project was mainly C++ code, but also included some Java and Python code and was on RHEL 7. We had Eclipse setup as the project's IDE, but I found that this IDE had limited performance and the offiically support version were outdated. Switching directly to VS Code also had performance problems we the VM got low on resources and we were limited to which versions we could run due to the old version of RHEL. The performance was better when using VS Code's Remote SSH extension to connect to the VM, but we due to version limitations on RHEL 7, we couldn't run a VS Code version that supported GitHub Copilot over Remote SSH extension.

The solution to solve this was to setup a docker container with a modern OS on the VM and then install a modern version of VS Code on the developer's host machine and connect to the container using the Remote SSH Extension.

The project was running on an old version of RHEL that didn't have support for modern versions of IDEs. To get around this, I setup a development environment using Docker and Visual Studio Code's Remote SSH extension.

Setup up a container like this wasn't too hard but did have a few special considerations. Port forwarding needed to be setup to allow the developer to SSH directly into the container. We used a ubuntu 24 container, which came with a default ubuntu user on user ID 1000. This needed to be deleted and a new user added to match the user account on the host. For the VS Code IDE to find header files when needed to give it some read-only access to project file and system files like /usr/include. We also needed to give the container access to some project specific tools and environment variables from the host. Finally, since the container was running Ubuntu 24, it had different versions of gcc/g++ so to build to needed to have the container ssh back to the host so some setup was needed to allow that.

To allow C++ intellisense to work inside VS Code we also needed to setup a JSON compilation database. To do this, another team member setup the Bear (Build EAR) tool. After this was done, myself and the team member who worked on the bear setup did a demo together for some of the other teams at the company to show them how to setup a similar environment for their projects.
